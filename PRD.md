## 一、产品背景

### 问题陈述

很多人（尤其有 ADHD 倾向或完美主义倾向的人）面临一个共同困境：

**想法很多，但启动很难。**

具体表现为：
- 念头和欲望在某些特定时刻大量涌现（洗澡、打扫、走路），但一拿起手机就消散
- 独自面对选择时，会陷入"寻找最优解"的循环，反而无法行动
- 有人陪伴时能自然启动，是因为注意力从碎片信息中抽离，无聊感出现，行动自然发生
- 计划和日程会触发"被安排感"，导致抗拒执行——哪怕是自己制定的计划

现有产品（body doubling 类、AI 日记类、任务管理类）都在解决"执行阶段"的问题，没有人在解决"启动"这一关键节点。

### 核心洞察

> "和别人在一起，即使走错了路也是共同的回忆；自己一个人的时候，就只是浪费。"

一个人行动之所以难以启动，不是缺乏意志力，而是**体验缺乏见证感**，错误只剩代价，没有价值。

产品要做的不是"激励"用户，而是：
1. 在念头最鲜活的时刻把它接住
2. 在合适的时机，用"邀请"而非"任务"的方式递回给用户

## 二、目标用户

**核心用户画像：**
- 有大量想法但执行力弱的人
- 对计划和提醒有抗拒感（"被安排感"）
- 容易陷入碎片信息，难以自然启动行动
- 有 ADHD 倾向、完美主义倾向，或处于低能量心理状态
- 独处时容易瘫痪，有人陪时能自然行动

**用户规模验证：**
- "Analysis paralysis"（分析瘫痪）和 body doubling 需求在神经多样性社群中有大量文献支持
- 2024 年研究显示 85% 的神经多样性用户通过 body doubling 显著改善了任务完成率
- 现有 body doubling 产品均聚焦于"执行陪伴"，未触及"启动"环节

## 三、产品定位

**产品名：** 念头

**一句话定位：**
> 一个收集你散落念头、在合适时机以朋友口吻邀请你行动的 AI 伴侣。

**不是什么：**
- ❌ 不是待办清单
- ❌ 不是日程管理工具
- ❌ 不是打卡 / 习惯追踪器
- ❌ 不是心理健康 / 冥想 app
- ❌ 不会提醒你"昨天没完成的事"

## 四、核心交互：双向滑动

### 设计哲学

产品的核心入口是一个录音按钮。用户按下说话，**说完之后通过滑动方向来表达意图**——向上滑表示"放入念头"，向下滑表示"取出念头"。

**为什么这样设计：**

传统的"双模式按钮"要求用户在按下录音前先做一次认知决定（"我现在是哪种模式"）。即使是自己选，也是一种微型的"被要求做决定"——违反了产品最核心的反抗拒原则。

而双向滑动让意图通过**身体动作**而非**认知判断**来表达：
- 用户按下录音直接说，不需要先想模式
- 说完后凭直觉滑动
- 方向的空间隐喻天然成立——上=收纳沉淀，下=打捞取出

这与"好奇心热身"的设计哲学一脉相承：**先动身体，认知会跟上。**

### 三个状态

**状态 1 · 静止**
- 录音按钮居中，上下色块均匀分布
- 上方为冷色调水彩（淡蓝、淡紫、淡绿），暗示"沉入"
- 下方为暖色调水彩（淡橙、淡粉、淡黄），暗示"绽放"
- 文字提示低调（45% 透明度），不抢视觉重心
- 上下色块做缓慢呼吸式起伏，无方向偏向

**状态 2 · 录音中**
- 录音按钮带温暖橙红色（#D85A30）的脉冲呼吸
- 两侧水彩开始流动，波纹分别向上向下推开
- 文字提示亮度提升，暗示"录完可以朝这两边去"

**状态 3 · 滑动中**
- 手指方向那一侧色块绽放扩展（约 60% 占比）
- 另一侧自然褪色（透明度降至 12%）
- 手指方向显示尾迹小点，强化"是我把它送过去"的感觉
- 主导侧文字提示扩大、加重

### 交互细节
- 录音方式：**按住录音，松手时已朝某方向滑了一段距离 → 直接触发对应模式**
- 不需要"先停止录音、再选择模式"的两步操作，一气呵成
- 滑动距离阈值需要在原型测试中调优

## 五、核心功能（v0.1 MVP）

### 功能 1：念头捕捉（向上滑）

**触发场景：**
1. **主动收纳：** 用户洗澡出来、打扫到一半、刷到某条内容突然心动——任何念头涌现的时刻
2. **清理干扰：** 用户正在做某件事（写作业、看书），脑子突然冒出不相关的念头（想查某个人的生平、想看一条新闻……）

两个场景在产品上是**同一个交互入口**，只是用户的使用心态不同。前者是"主动收纳"，后者是"清理干扰"——但都通过按住录音 + 向上滑来完成。

**设计原则：**
- 极低摩擦：一键打开，**语音作为主入口**，文字输入作为次选
- 无需整理、无需分类，直接扔进去
- 支持各种类型的念头：想去的地方、想做的事、想看的东西、当下情绪、分心的想法

**交互细节：**
- 主界面正中是录音按钮，按住说话，向上滑松手
- 录音按钮下方提供"打字"小入口，给不方便说话的场景兜底
- 录音结束后直接存入，不显示转录结果让用户检查（避免触发完美主义编辑欲）
- 反馈极简："记下了。"

**AI 后台处理（静默）：**
- 提取关键信息：地点、活动类型、情绪色彩、时间相关性（"周末"、"天还亮着"）
- 评估念头温度（新鲜度）
- 存入念头池，不展示给用户

---

### 功能 2：状态问询 + AI 邀请（向下滑）

**触发场景：** 用户主动打开 App，希望"现在有点什么可做"。

**流程：**
1. 用户按住录音，简单说一句当下状态（"有点闷" / "想动一动" / "无聊"）
2. 向下滑松手 → 进入邀请模式
3. AI 从念头池中选出一个最匹配的念头，发出邀请
4. 用户三选一回应

**示例邀请：**
> "今天天还亮着——你之前说想去那个媒体图书馆借乐器，走过去也就十分钟，要不要去看看？"

**用户回应方式（三选一）：**
- 快速按钮："好，去看看" / "今天先这样"
- 语音回应：长按说话
- 文字回应：自由输入

提供多种回应方式，是因为用户当下的状态不同——情绪低时按钮就够了，有话想说时可以自由表达。

**示例拒绝响应：**
> "好，那今天先这样。"（对话结束，无追问）

## 六、UI 设计语言

### 视觉基调
**明亮、流动、干净、水彩画质感的疗愈系浅色风格。**

视觉语言和产品气质保持一致——水彩的流动感本身传递"不强迫"的感觉，与"邀请而非命令"的产品原则在视觉上相互呼应。

### 背景与色彩
- **背景：** 接近纸张的暖白色（#FBF9F4），不是纯白，带有轻微的暖调
- **色彩：** 淡而流动的水彩色块，作为念头捕捉界面和对话背景的装饰层——不是图标或按钮的颜色，而是界面的"气氛层"
- **冷色调（上方/记录）：** 淡蓝、淡紫、淡绿
- **暖色调（下方/邀请）：** 淡橙、淡粉、淡黄
- 所有色块低饱和度，像未干的水彩在纸上晕开

### 字体与排版
- **AI 文字：** 衬线字体，传递"短信"或"便条"的感觉，而非系统提示
- **用户输入和按钮：** 简约无衬线
- 字号克制，行距宽松，文字量少
- 不用粗体强调，不用图标堆砌

### 交互感
- 动效轻柔、缓慢，像水彩晕染而非弹窗跳出
- 念头存入时有一个轻微的"沉入"感，不是勾选完成的满足感
- 邀请出现时像一张便条自然出现，不是通知横幅

## 七、技术架构

### 产品形态
**PWA（Progressive Web App）**
- 可添加到手机主屏幕，体验接近原生 App
- 无需上架 App Store，开发成本低

### 技术栈
| 层级 | 技术选型 |
|------|----------|
| 前端 | React + PWA（Vite） |
| 后端 | Python + FastAPI |
| AI | Claude API（claude-haiku-4-5） |
| 数据库 | SQLite |
| 开发工具 | Claude Code |

**为什么用 Haiku 4.5 而非 Sonnet：** 本产品的 AI 任务相对简单（提取信息、选念头、生成一句邀请），不需要更高级别的推理能力。Haiku 4.5 价格更低（$1 / $5 每百万 tokens），响应更快，足以胜任。

### 数据模型（核心）
```
念头（Thought）
- id
- content_raw        # 原始输入（语音转文字或文字）
- extracted_info     # AI 提取的结构化信息（地点、活动、情绪、时间）
- temperature        # 新鲜度权重（随时间衰减）
- created_at
- last_invited_at    # 上次被邀请的时间
- decline_count      # 被拒绝次数
- status             # active / snoozed / archived

对话记录（Conversation）
- id
- date
- state_checkin      # 状态问询的用户回答
- invited_thought_id # 本次邀请的念头
- user_response      # 用户对邀请的回应
- outcome            # accepted / declined / ignored
```

### Token 成本控制
- 每次对话 AI 调用：状态问询 + 邀请生成，预计 200-400 tokens
- 念头捕捉：提取结构化信息，预计 100-200 tokens
- 每日调用次数：用户主动触发，无后台自动高频调用
- 预计单用户每日成本：< $0.005（Haiku 4.5 价格下）

## 九、念头生命周期与排序算法

### 综合权重公式

```
权重 = 新鲜度衰减 × 冷却状态 × AI 上下文匹配分
```

前两项是规则计算（无 token 消耗），第三项交给 LLM 做语义判断。

### 维度 1：新鲜度衰减

```
新鲜度 = e^(-Δt / τ)
```

- `Δt` ：念头记录至今的时间
- `τ` ：衰减半衰期，MVP 阶段设为 **7 天**

### 维度 2：邀请冷却

| 用户反应 | 冷却天数 |
|---------|---------|
| 接受邀请 | 14 天 |
| 明确拒绝 | 5 天 |
| 无回应 | 3 天 |

### 维度 3：AI 上下文匹配

发邀请时，传给 Claude 的 prompt 包含：
- 当前时间（几点、星期几）
- 用户当下的状态问询回答
- 池子里所有还在活跃期的念头列表

Claude 的任务：从活跃念头中选出**最匹配当前上下文**的一个。

### 念头的归档

念头不会永远活着，以下情况自动归档：
- 衰减到原始权重 5% 以下（约 21 天未被选中）
- 被明确拒绝 3 次

## 十、设计原则

1. **低门槛优先**：任何交互超过 3 步就需要重新审视
2. **邀请而非命令**：语气永远是朋友，不是 coach、不是提醒系统
3. **拒绝是正常选项**：declined 不是失败，不触发任何负面反馈
4. **不展示积压**：用户永远看不到"你还有 12 个未完成的想法"
5. **念头有生命周期**：自然衰减，不永远等你
6. **身体先于认知**：所有关键决策通过身体动作（按住、滑动、点击卡片）而非认知判断（选择模式、勾选选项）来完成

## 十一、版本范围

### v0.1 MVP（一周内完成）

**包含：**
- 双向滑动核心交互（向上记录 / 向下邀请）
- 录音 + 语音转文字
- 念头存储和排序算法
- AI 选念头 + 生成邀请
- 用户对邀请的三种回应方式
- 水彩 UI 风格

**目标：** 能完整演示产品的核心创新（滑动交互 + AI 邀请机制），UI 精致，能拿给面试官看。

### v0.2 计划（MVP 之后）

明确列入下一版规划，但 v0.1 不做。这些功能在 PRD 里设计完整，是为了展示完整的产品思考，不是为了短期内全部实现。

- **好奇心热身**：紧要事项 / 启动困难场景下的步骤拆解对话
- **步骤完成的随机反馈机制**：乐器音频 + 视觉彩蛋的素材库（变量奖励）
- **归档区**：藏在二级页面的念头回看入口
- **首次使用引导动画**：演示双向滑动的 10 秒动画
- **回复语料库**：所有 AI 给用户的"系统级回应"（如"记下了"、邀请回应、状态确认等）使用语料库随机选择，而非固定一句。例如"记下了"可以是"听起来不错"、"什么时候去看看吧"、"嗯，先放在这里"。设计意图：避免重复机械感，增加"像朋友"的真实感。

### v0.3+ 后续规划

- 推送通知策略
- 用户账号系统
- 对话历史的日记式回看

### v0.1 明确不做

- ❌ 社交 / 分享功能
- ❌ 数据分析 / 情绪趋势图
- ❌ 与日历同步
- ❌ 多语言支持

## 十三、简历亮点总结

这个项目展示的产品思维：

- **从用户心理机制出发**（analysis paralysis、body doubling 效应、被安排感）而非功能堆砌
- **明确的反设计**（不做提醒、不做清单、不做打卡）——能说清楚不做什么，比能说做什么更重要
- **AI 的角色是"判断时机"，不是"生成内容"**——这是更难也更有价值的 AI 产品设计
- **成本可控的 AI 架构**（用户主动触发、轻量 prompt、token 预算清晰、规则与 LLM 分工明确）
- **身体先于认知的交互哲学**——双向滑动取代模式选择，是把心理学洞察转化为交互设计的具体例证
- **变量奖励的精准使用**（v0.2 规划）——知道什么时候用、什么时候不能用（奖励身体动作而非任务完成），体现对心理机制副作用的清醒认知
- **清晰的版本范围控制**——v0.1 只做核心链路，v0.2 加深度功能，体现对"做什么"和"先不做什么"的判断
- **细节层面的产品意识**——比如发现"固定回复"会让产品变得机械，主动提出回复多样化方案。这种对微小体验的敏感度是产品思维成熟度的标志。
